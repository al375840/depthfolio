# Azure Static Web Apps deployment

## What and why

Azure Static Web Apps (SWA) is Microsoft's managed platform for SPA + linked-API deployments. The Hobby tier is free and includes:

- Global CDN with HTTPS by default.
- Custom domain with automatic Let's Encrypt certificates.
- Linked Azure Functions API (one per SWA instance), invoked from the same origin as the frontend (no CORS).
- GitHub Actions CI/CD generated automatically when you link a repo.
- Preview environments for every pull request.

## Provisioning

In the Azure Portal:

1. Create resource → Static Web App.
2. Plan: **Free (Hobby)**. Region: **West Europe** (or nearest).
3. Source: GitHub. Authorise and pick your repo + `main` branch.
4. Build presets: **Custom**. Set:
   - `app_location`: `apps/web`
   - `api_location`: `apps/api/src/Portfolio.Api`
   - `output_location`: `dist/web/browser`

Azure automatically:

- Creates `.github/workflows/azure-static-web-apps-<random>.yml`.
- Adds `AZURE_STATIC_WEB_APPS_API_TOKEN_<RANDOM>` as a repository secret.

Commit the generated YAML; never edit the secret in the file (use the secret reference).

## staticwebapp.config.json

Lives at the repo root. Controls routing, headers, MIME types, auth, and fallback.

```json
{
  "navigationFallback": {
    "rewrite": "/index.html",
    "exclude": ["/api/*", "/models/*", "/fallback/*", "/*.{onnx,bin,png,jpg,jpeg,webp,glsl}"]
  },
  "globalHeaders": {
    "Cross-Origin-Opener-Policy": "same-origin",
    "Cross-Origin-Embedder-Policy": "require-corp",
    "X-Content-Type-Options": "nosniff",
    "X-Frame-Options": "DENY",
    "Referrer-Policy": "strict-origin-when-cross-origin"
  },
  "routes": [
    {
      "route": "/models/*",
      "headers": {
        "Cache-Control": "public, max-age=31536000, immutable"
      }
    },
    {
      "route": "/fallback/*",
      "headers": {
        "Cache-Control": "public, max-age=31536000, immutable"
      }
    }
  ],
  "mimeTypes": {
    ".onnx": "application/octet-stream",
    ".glsl": "text/plain"
  },
  "responseOverrides": {
    "404": { "rewrite": "/index.html", "statusCode": 200 }
  }
}
```

## GitHub Actions workflow

Generated by Azure, but here's the shape so you can edit it confidently:

```yaml
name: Azure Static Web Apps CI/CD

on:
  push:
    branches: [main]
  pull_request:
    types: [opened, synchronize, reopened, closed]
    branches: [main]

jobs:
  build_and_deploy:
    if: github.event_name == 'push' || (github.event_name == 'pull_request' && github.event.action != 'closed')
    runs-on: ubuntu-latest
    name: Build and deploy
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: true
          lfs: false
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'
      - name: Build and deploy
        id: builddeploy
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }}
          repo_token: ${{ secrets.GITHUB_TOKEN }}
          action: 'upload'
          app_location: 'apps/web'
          api_location: 'apps/api/src/Portfolio.Api'
          output_location: 'dist/web/browser'

  close_pr_preview:
    if: github.event_name == 'pull_request' && github.event.action == 'closed'
    runs-on: ubuntu-latest
    steps:
      - uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }}
          action: 'close'
```

## Application settings (secrets in production)

Frontend env vars at build time: edit `.github/workflows/...yml` and set them under `env:` for the deploy step.

Backend secrets (used by Functions at runtime): Azure Portal → your Static Web App → Configuration → Application settings. Examples:

| Key | Example value |
|---|---|
| `Resend__ApiKey` | `re_xxx_...` |
| `Contact__DestinationEmail` | `sympoleon@gmail.com` |
| `Contact__FromAddress` | `noreply@depthfolio.dev` |

Double-underscore is the convention for nested JSON sections in .NET (`Resend:ApiKey` becomes `Resend__ApiKey` in env vars).

## Custom domain

1. Buy the domain (Namecheap, Cloudflare Registrar, Porkbun, etc.).
2. In Azure Portal → your SWA → Custom domains → Add.
3. Azure gives you a TXT record to add at your DNS provider (validation) and either an ALIAS/CNAME or A records to point traffic.
4. Wait for DNS propagation (5–60 min).
5. Azure issues the Let's Encrypt cert automatically.

## Gotchas

- **`navigationFallback` only triggers on 404s** for routes the CDN doesn't recognise. If your route doesn't match any file and isn't excluded, it falls back to `index.html`. Exclude API paths and asset extensions.
- **`Cross-Origin-Embedder-Policy: require-corp` breaks third-party scripts** that don't send `Cross-Origin-Resource-Policy` headers. If you add an external script (analytics, etc.), check it supports CORP or host it yourself.
- **The Free tier has limits**: 100 GB bandwidth/month, 0.5 GB storage. Plenty for a portfolio; not for a viral product.
- **Functions cold start is ~1s on the Free tier**. The first contact-form submit of the day will feel slow. Acceptable here.
- **`output_location` is relative to `app_location`**, not the repo root. Get this wrong and the deploy uploads nothing useful.
- **PR previews share the production database/secrets** by default. For depthfolio there are no secrets to leak, but be aware on bigger projects.

## Verifying after deploy

After the first deploy:

1. `https://<random>.azurestaticapps.net` should serve the Angular app.
2. `https://<random>.azurestaticapps.net/api/contact` should return 405 for GET and 202 for valid POST.
3. Browser devtools → Network → check response headers include `Cross-Origin-Opener-Policy: same-origin`.
4. Run `crossOriginIsolated` in the JS console — should be `true`. If `false`, SharedArrayBuffer won't work.
